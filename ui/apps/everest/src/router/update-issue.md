Недавно мы реализовали overview page, где отображаем структуру UI-схемы с разбиением на секции. Если схема содержит несколько sections, они маппятся и отображаются на overview-форме.

Проанализируй пожалуйста main-ветку и поведение редактирования, и предложи архитектурный план реализации. Каждая секция должна редактироваться отдельно через модальное окно, в котором отображаются только поля этой секции. Далее будут требования:

1. Валидация
   Несмотря на то, что редактируется только одна секция, валидация должна выполняться для всей формы.
   Необходимо учитывать зависимости между полями (например, cell expressions), даже если связанные поля не отображаются в текущей модалке.
   Для проверки должна использоваться вся UI-схема (на столько, на сколько это имеет смысл. нет никакого смысла прогонять zod валидацию для полей из другой секции, но вот cell expressions которые могут быть перекрестными между секциями важны)

2. Предзаполнение данных
   При открытии модального окна поля должны быть заполнены значениями из instance (загруженного на dbOverview page), а не дефолтами из схемы. Так же пожалуйста проанализируй как это делалось ранее, уверена, что большинство логики и оберток останется таким же.

3. Переиспользование и архитектура
   Нужно максимально переиспользовать существующие компоненты UI-генератора и новые, если ты будешь их создавать.
   Если текущая реализация не покрывает кейс — расширять её через режимы, а не дублировать логику.

4. Режимы формы
   В системе уже есть и будут добавляться разные режимы (опять таки ты можешь подглядеть в main что они из себя представляют)

Create — создание новой сущности
Edit — редактирование существующей
Restore — создание с предзаполненными данными из backup
Import — появится позже

Архитура должна учитывать эти режимы как базовую концепцию.

5. Управление поведением через схему
   Пользователь должен иметь возможность задавать поведение полей в UI-схеме, например

disabled

Это должно зависеть от режима формы (например, поле может быть доступно в Create, но disabled в Edit).

Предлагаю реализовать это через отдельные конфигурации для режимов (например: edit, restore, import), внутри которых можно задавать поведение полей. Но ты можешь предложить и свою архитектуру.
